1. Armazenamento do Token

A arquitetura utiliza dois tokens com estratégias de armazenamento distintas, seguindo o padrão Split Token:

Access Token — sessionStorage

O access token JWT é armazenado em sessionStorage via AutenticacaoStore:

// autenticacao.store.ts
const SESSION_KEY_TOKEN = 'pj_access_token';
sessionStorage.setItem(SESSION_KEY_TOKEN, token);
  • Escopo da aba — desaparece ao fechar o browser
  • Acessível via JavaScript — vulnerável a XSS
  • Não persiste entre abas ou após reinicialização

Refresh Token — HttpOnly Cookie

O refresh token opaco é gerenciado exclusivamente pelo backend:

// CookieHelper.cs
new CookieOptions {
    HttpOnly = true,
    Secure   = !env.IsDevelopment(),     // HTTPS em produção
    SameSite = SameSiteMode.Strict,      // bloqueia cross-site em prod
    Expires  = DateTimeOffset.UtcNow.AddDays(7),
}
  • Inacessível ao JavaScript — protegido contra XSS
  • Enviado automaticamente pelo browser em todas as requisições
  • SameSite=Strict em produção bloqueia CSRF
⚠
Risco R1 — Alta: O access token em sessionStorage é legível por qualquer script JavaScript na mesma origem. Um ataque XSS bem-sucedido pode exfiltrar o token e usá-lo por até 1 hora (tempo de expiração).

2. Content Security Policy (CSP)

O index.html não contém <meta http-equiv="Content-Security-Policy"> e o backend não emite o header Content-Security-Policy nas respostas HTTP.

⚠
Risco R2 — Alta: Sem CSP, um XSS injetado pode carregar scripts externos, acessar sessionStorage e exfiltrar o access token para um servidor remoto sem qualquer restrição de origem.

Recomendação

Adicionar ao backend (middleware) ou ao index.html:

// backend.cs — adicionar no pipeline
app.Use(async (ctx, next) => {
    ctx.Response.Headers["Content-Security-Policy"] =
        "default-src 'self'; " +
        "script-src 'self'; " +
        "style-src 'self' https://fonts.googleapis.com; " +
        "font-src 'self' https://fonts.gstatic.com; " +
        "connect-src 'self'; " +
        "frame-ancestors 'none';";
    await next();
});

Tipo JWS (apenas assinado) Payload não criptografado — legível por qualquer um com o token
Algoritmo HS256 (HMAC-SHA256) Chave simétrica — mesmo segredo para assinar e validar
Comprimento mínimo da chave 32 caracteres (verificado no código) Aceito, mas recomenda-se ≥ 64 bytes aleatórios

// JwtService.cs
var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(secret));
_signingCredentials = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
ℹ
Risco R3 — Média: HS256 usa chave simétrica. Se o Jwt:Secret vazar (logs, dump de memória, variável de ambiente exposta), qualquer pessoa pode forjar tokens arbitrários. RS256 (assimétrico) isolaria a capacidade de assinatura da de validação.

4. Tempo de Vida e Renovação

TokenTTL (appsettings.json)Comportamento
Access Token 1 hora JWT assinado, válido imediatamente após emissão
Refresh Token 7 dias Opaco (64 bytes aleatórios), armazenado em memória no servidor

Fluxo de Renovação

  1. Access token expira → interceptor Angular recebe HTTP 401
  2. Interceptor chama automaticamente POST /auth/refresh
  3. Browser envia o refresh token via HttpOnly Cookie (sem acesso JavaScript)
  4. Backend valida o refresh token, revoga o antigo e emite novo (rotação)
  5. Novo access token retorna no body JSON → salvo em sessionStorage
  6. Requisição original é repetida com o novo token
  7. Se refresh falhou → store.limpar() + redirect para /login
// autenticacao.interceptor.ts — lógica de retry após 401
catchError((erro: HttpErrorResponse) => {
  if (erro.status === 401 && !isAuthEndpoint) {
    return authService.refresh().pipe(
      switchMap(response => {
        const reqRetry = req.clone({
          withCredentials: true,
          setHeaders: { Authorization: `Bearer ${response.accessToken}` },
        });
        return next(reqRetry);
      }),
      catchError(erroRefresh => {
        store.limpar();
        router.navigate(['/login']);
        return throwError(() => erroRefresh);
      }),
    );
  }
  return throwError(() => erro);
})
✅
Controle P1 — Mitigado: A rotação de refresh token está implementada. Cada uso do refresh token emite um novo e invalida o anterior, mitigando ataques de replay se o token for interceptado.

5. Revogação de Tokens

Access Token

O access token JWT não possui mecanismo de revogação antecipada. Uma vez emitido, permanece válido até seu exp (1 hora), independentemente de logout ou comprometimento da conta.

⚠
Risco R4 — Média: Um access token roubado pode ser usado por até 1 hora sem possibilidade de bloqueio imediato. Não existe blacklist ou JTI denylist implementado.

Refresh Token

O refresh token possui revogação imediata via RefreshTokenStore.Revogar():

CenárioMecanismoStatus
Logout voluntário POST /auth/logout revoga token + deleta cookie ✅ Implementado
Uso do refresh token Token antigo revogado antes de emitir novo (rotação) ✅ Implementado
Token expirado Verificação de entry.Expiracao < DateTime.UtcNow ✅ Implementado
Reinicialização do processo Store in-memory — todos os tokens são perdidos ⚠ Gap
ℹ
Risco R5 — Média: O RefreshTokenStore é in-memory (ConcurrentDictionary). Um restart do processo invalida todas as sessões ativas (logout involuntário de todos os usuários). Em produção, substituir por Redis ou tabela de banco com TTL, conforme indicado no próprio código-fonte.

7. Escopo, Permissões e Dados Sensíveis

Payload do Access Token (decodificável sem a chave secreta)

{
  "sub":                  "12345",           // matrícula do funcionário
  "jti":                  "uuid-v4",         // ID único do token (não validado em blacklist)
  "http://...nameid":     "12345",           // ClaimTypes.NameIdentifier
  "http://...name":       "João da Silva",   // ClaimTypes.Name
  "http://...role":       "GERENTE",         // ClaimTypes.Role — EXPOSTO
  "nbf": 1747872000,
  "exp": 1747875600,                         // +1 hora
  "iss": "plataforma-pj",
  "aud": "plataforma-pj-client"
}
ClaimConteúdoRisco
sub Matrícula do funcionário Baixo
name Nome completo do funcionário Baixo
role Perfil de acesso (ex: GERENTE) Médio — enumeração de perfis
Dados de cliente Nenhum (CPF, CNPJ, saldo — não incluídos) ✅ Correto
ℹ
Risco R6 — Baixo: O campo role (perfil do funcionário) é visível por qualquer pessoa que possua o token, bastando decodificar o payload em Base64. Não há dados sensíveis de clientes no token — o que é correto.

8. Controles Adicionais Identificados

✅
Rate Limiting no Login: 5 requisições por minuto por IP. Mitiga força bruta e credential stuffing.
✅
Bcrypt nas Senhas: Work factor 12 em produção (4 em dev). Passwords nunca armazenadas em texto claro.
✅
Audit Log: Logins bem-sucedidos, inválidos e rotações de refresh token são registrados com matrícula e IP ([AUDIT]).
✅
Sem enumeração de usuários: Login inválido retorna mensagem genérica "Credenciais inválidas" sem indicar qual campo está errado.

9. Resumo Executivo de Riscos

# Risco Severidade Status Recomendação
R1 Access token em sessionStorage — vulnerável a XSS Alta Ativo Migrar para cookie HttpOnly ou reduzir TTL para ≤ 15 min com CSP
R2 Sem Content Security Policy — XSS pode exfiltrar token Alta Ativo Implementar CSP restritivo no backend (header) e no index.html
R3 HS256 (chave simétrica) — vazamento do secret compromete tudo Média Ativo Avaliar migração para RS256; rotacionar secret periodicamente
R4 Access token irrevogável — válido por até 1h após comprometimento Média Ativo Implementar JTI denylist (Redis) ou reduzir TTL para 15 min
R5 RefreshTokenStore in-memory — não sobrevive restart Média Ativo Substituir por Redis ou banco de dados com TTL em produção
R6 role exposto no payload do token (legível sem a chave) Baixa Ativo Avaliar uso de JWE (criptografia do payload) ou remover claim de role
Controles mitigados: Rotação de refresh token ✅ | HTTPS + HSTS ✅ | Rate limiting ✅ | Bcrypt ✅ | CORS restrito ✅ | Audit log ✅
